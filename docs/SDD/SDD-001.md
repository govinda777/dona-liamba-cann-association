# SDD-001 – Refinamento Técnico: Jornada de Consulta, Pagamentos e Integração Memed

- **Status:** Proposto
- **Data:** 2026-03-06
- **Referência:** [ADR-001](../ADR/ADR-001.md)
- **Autor:** Jules (Agent)

## 1. Introdução

Este Documento de Design de Software (SDD) detalha a implementação técnica das decisões tomadas no ADR-001. O objetivo é descrever a arquitetura de serviços, esquemas de dados e integrações necessárias para suportar a jornada de consulta, pagamentos (Fiat/Cripto) e a sincronização automática com a Memed.

## 2. Arquitetura de Serviços (Backend)

O backend será estruturado em quatro serviços principais (módulos lógicos dentro do Next.js API Routes/Actions):

### 2.1 SCHED (Scheduling Service)
Responsável pela gestão da agenda dos médicos e agendamentos de pacientes.
- **Responsabilidades:** Criação de slots, reserva de horários, gestão de status do agendamento (pendente, pago, cancelado).

### 2.2 PAY (Payment Service)
Gerencia o fluxo de pagamentos multi-rail.
- **FIAT/PIX:** Integração com gateway (PSP) para geração de QR Codes e processamento de webhooks.
- **Cripto:** Interface com smart contracts de Escrow e Smart Wallets (via Privy/Wagmi/Viem).
- **Split:** Lógica de cálculo de comissão (2% conforme `EscrowMarketplace.sol`) e repasse.

### 2.3 SESSION (Consultation Session Service)
Gerencia a sala virtual de consulta.
- **Integração:** Jitsi Meet ou Twilio Video.
- **Tokens:** Geração de links JWT com permissões específicas para médico e paciente.
- **Logs:** Registro de eventos de presença para o `AttendanceLog`.

### 2.4 DOCS (Sync & Documents Service)
Responsável pela integração com a Memed e gestão de metadados.
- **Polling:** Job assíncrono para busca de novas prescrições.
- **Matching:** Algoritmo de vinculação automática de prescrição à consulta.

## 4. API Design (Endpoints Principais)

### Scheduling
- `POST /api/v1/slots`: Médico cria disponibilidade.
- `GET /api/v1/slots?doctor_id={id}`: Paciente consulta agenda.
- `POST /api/v1/bookings`: Reserva de horário.

### Payments
- `POST /api/v1/payments/pix`: Gera cobrança PIX.
- `GET /api/v1/payments/crypto/status/{txHash}`: Verifica status on-chain.
- `POST /api/v1/payments/webhook`: Callback do gateway FIAT.

### Sessions
- `GET /api/v1/sessions/{id}/join`: Retorna token de acesso à sala.
- `POST /api/v1/sessions/{id}/end`: Médico finaliza consulta manualmente.

## 5. Algoritmo de Matching (Memed)

O serviço `DOCS` executará o seguinte fluxo:

1. **Trigger:** Consulta movida para status `REALIZADA`.
2. **Delayed Job:** Aguarda 5 minutos e inicia polling na API Memed.
3. **API Call:**
   ```
   GET /prescricoes?medico_crm={CRM}&data_inicio={hora_inicio_consulta - 1h}
   ```
4. **Filtro:**
   - Busca por prescrições cujo `patient_cpf` coincida com o CPF do paciente da consulta.
   - Seleciona a prescrição criada no intervalo [hora_inicio_consulta, hora_inicio_consulta + 48h].
5. **Persistence:** Salva em `PrescriptionMetadata` e notifica usuários.

## 6. Segurança e Conformidade

- **Criptografia:** Metadados sensíveis em `PrescriptionMetadata` (como CPF se armazenado) devem ser criptografados.
- **Assinatura Digital:** A plataforma verificará se o `document_type` exige assinatura e o status da mesma via Memed.
- **LGPD:** Termo de consentimento específico para compartilhamento de metadados de prescrição entre médico, paciente e plataforma.
